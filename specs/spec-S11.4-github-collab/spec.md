# Spec S11.4 — GitHub Collaboration

## Overview
Establishes and verifies the GitHub collaboration hygiene practices for the project: branch protection on `main`, a consistent branch-naming convention (`spec/SX.Y-name`), PR-per-spec workflow with mandatory review, and a commit-message convention (`SX.Y(impl): …`). Both students must have visible contributions. This spec ensures the repository reflects real team collaboration for the graded deliverable.

## Dependencies
- S1.1 (dependency declaration — project must exist and be committed)

## Target Location
- `.github/` (branch protection rules, PR templates, optional CODEOWNERS)
- GitHub repository settings (branch protection UI or `gh` CLI)

---

## Functional Requirements

### FR-1: Protected `main` branch
- **What**: The `main` branch must be configured with branch protection that prevents direct pushes and requires at least one pull request review before merging.
- **Inputs**: GitHub repository settings (configured via the GitHub UI or `gh api`).
- **Outputs**: Any attempt to push directly to `main` is rejected by GitHub. PRs to `main` require at least 1 approved review.
- **Edge cases**: Repo admin bypasses are acceptable for emergency fixes but must be documented in the PR description; force-push to `main` must remain disabled.

### FR-2: Branch naming convention
- **What**: All feature/spec work must live on branches named `spec/SX.Y-<slug>` (e.g. `spec/S5.1-apogee-perigee-sieve`). The two long-lived student branches (`feature/satcollision-nishant`, `feature/satcollision-<name>`) are grandfathered in.
- **Inputs**: Branch names in the repository.
- **Outputs**: Every spec-implementing branch follows the `spec/SX.Y-<slug>` pattern; no spec work lands directly on `main` or unnamed scratch branches.
- **Edge cases**: Hotfix branches may use `hotfix/<desc>` if documented; doc-only changes may use `docs/<desc>`.

### FR-3: PR-per-spec with review
- **What**: Each spec (`SX.Y`) is delivered via exactly one PR. The PR title references the spec ID and slug. The PR must receive at least one review comment or approval from the other student before merge.
- **Inputs**: Open/merged PRs on the repository.
- **Outputs**: PR titles match `SX.Y(<type>): <description>` pattern (e.g. `S5.4(impl): TCA refinement & miss distance`). Each PR's description cites the spec it implements and links the `checklist.md`.
- **Edge cases**: A spec that spans multiple commits is fine; the PR is the unit of delivery, not the commit. A single PR may cover closely related sub-items only if noted in the spec.

### FR-4: Commit message convention
- **What**: Commits follow `SX.Y(<type>): <short description>` where `<type>` is one of `spec`, `impl`, `fix`, `test`, `docs`, `refactor`.
- **Inputs**: Git commit messages in the repository.
- **Outputs**: No commit on `main` (post-merge) has a generic message like "fix", "update", or "wip". Squash-merge or rebase-merge are both acceptable as long as the resulting commit carries the convention.
- **Edge cases**: Merge commits generated by GitHub (`Merge pull request #N`) are exempt. Initial bootstrapping commits are exempt.

### FR-5: Both students have visible contributions
- **What**: Both students must have commits (or co-authored commits) merged to `main` covering their designated ownership areas (S1 = backend; S2 = frontend).
- **Inputs**: `git log --all --author` output for both email addresses.
- **Outputs**: At least 5 commits from each student merged to `main`. Contributions are traceable by author email.
- **Edge cases**: Pair-programmed sessions may use `Co-Authored-By:` trailers. A student who only reviews (no author credit) does not satisfy this requirement.

### FR-6: PR template
- **What**: A `.github/pull_request_template.md` exists that prompts contributors to fill in the spec ID, checklist link, test status, and reviewer notes.
- **Inputs**: The `.github/` directory.
- **Outputs**: The file exists and contains fields for: Spec ID, Checklist link, Tests passing (checkbox), Lint passing (checkbox), Notes for reviewer.
- **Edge cases**: Template is a guide, not enforced by GitHub automation; reviewers should flag PRs that skip required fields.

## Test-Driven Requirements

### Tests to Write First (Red → Green)

Because this spec governs repository configuration and process (not runtime code), the "tests" are a mix of automated script checks and manual verification steps:

1. **test_pr_template_exists**: Assert `.github/pull_request_template.md` exists and contains the required section headings (`Spec ID`, `Checklist`, `Tests passing`, `Lint passing`).
2. **test_commit_messages**: Parse `git log --format="%s" main` and assert that non-exempt commits match the regex `^S\d+\.\d+\([a-z]+\): .+` or are merge-commit exemptions.
3. **test_both_authors_contributed**: Run `git log --format="%ae" main` and assert that both student email addresses appear at least 5 times.
4. **test_branch_naming**: Run `git branch -a` and assert that all non-main, non-grandfathered branches match `spec/S\d+\.\d+-` or `feature/satcollision-` or `hotfix/` or `docs/`.

### Mocking Strategy
- No external HTTP mocks needed — all checks run against the local git repository and the filesystem.
- GitHub branch-protection rules cannot be tested in pytest; they are verified manually via the GitHub UI or `gh api repos/{owner}/{repo}/branches/main/protection`.
- Tests live in `backend/tests/test_github_collab.py` (or a dedicated `tests/` root-level file).

### Coverage Expectation
- All 4 automated checks have passing tests.
- Manual checklist covers branch protection and PR review requirements.
